Entities

The integration exposes nine entity platforms. Most entities are backed by Device API v2 endpoints polled every 30 seconds. The table below gives a quick overview; each platform is described in detail below.

Platform Entity key(s) Model API endpoint(s)
Button next_app, previous_app, dismiss_current_notification, dismiss_all_notifications All PUT /apps/next, /apps/prev, DELETE /notifications/current
Light sky_light SKY only GET/PUT /api/v2/device/display
Number brightness, volume non-SKY / All PUT /api/v2/device/display, PUT /api/v2/device/audio
Scene one per widget All PUT /api/v2/device/apps/:id/widgets/:wid/activate
Select brightness_mode non-SKY PUT /api/v2/device/display
Sensor rssi All GET /api/v2/device (polled)
Switch display_on, bluetooth_active All (availability-gated) PUT /api/v2/device/display, PUT /api/v2/device/bluetooth
Text bluetooth_name All (Bluetooth gated) PUT /api/v2/device/bluetooth
Update firmware All (OS ≥ 2.3.0) GET /api/v2/device (polled)

Button

PUT /api/v2/device/apps/next API 2.1.0+
PUT /api/v2/device/apps/prev API 2.1.0+
DELETE /api/v2/device/notifications/current API 2.0.0+

Four stateless button entities are created per device. Pressing a button fires a single API call and does not have a persistent state.

Entity key Icon Action API call
next_app mdi:skip-next Advance to the next installed app PUT /api/v2/device/apps/next
previous_app mdi:skip-previous Return to the previous installed app PUT /api/v2/device/apps/prev
dismiss_current_notification mdi:bell-off Dismiss the notification currently shown on screen DELETE /api/v2/device/notifications/current
dismiss_all_notifications mdi:bell-remove Clear the entire notification queue DELETE /api/v2/device/notifications (all)

Unique IDs follow the pattern <serial>-<key> (e.g. SA2301234567-next_app).

See the Apps and Notifications API references for endpoint details.

Light SKY only

GET /api/v2/device/display API 2.0.0+
PUT /api/v2/device/display API 2.0.0+

One light entity is created for every LaMetric SKY device. It represents the full LED matrix display and exposes on/off control and brightness adjustment in a single entity. Non-SKY devices (e.g. LaMetric Time) do not get a light entity; their brightness is controlled through the Number and Select entities instead.

Supported features

Feature Details
Color mode BRIGHTNESS — single white channel, 0–255 mapped to the device's brightness range (reported by brightness_limit)
On / Off Sends {"on": true/false} to PUT /api/v2/device/display
Brightness HA 0–255 scale is mapped to the device's brightness_range (min to max %) via a linear scale. The effective cap is given by brightness_limit.max (may be lower than the range max when powered via USB).
Streaming host The Light entity is also the target for the pixel streaming entity services (start_stream, send_stream_data, stop_stream). See the Pixel Streaming page for full details.

Unique ID: <serial>-sky_light. See the Display API reference.

Number

PUT /api/v2/device/display API 2.0.0+
PUT /api/v2/device/audio API 2.0.0+

Up to two number entities are created per device, subject to hardware availability reported by the device state.

Entity key Unit Range Model API field
brightness % Reported by brightness_limit (min–max) non-SKY only display.brightness → PUT /display
volume % Reported by volume_range (typically 0–100) All (if audio available) audio.volume → PUT /audio

Both entities are placed in the Configuration entity category. The step size is fixed at 1. The range is not hard-coded: it is read from the device state on each coordinator refresh, so it automatically reflects any hardware limits (e.g., USB power cap on brightness).

brightness vs. brightness_mode Setting brightness while brightness_mode is auto is accepted by the API but has no visible effect until brightness_mode is switched to manual. Refer to the Display API reference.

See also the Audio API reference.

Scene

GET /api/v2/device/apps API 2.1.0+
PUT /api/v2/device/apps/:package/widgets/:id/activate API 2.1.0+
POST /api/v2/device/apps/:package/widgets/:id/actions API 2.1.0+

One scene entity is created for every installed app widget on the device. The list of apps is fetched from GET /api/v2/device/apps and updated on each coordinator refresh. New widgets (e.g., added via the LaMetric app) are picked up automatically without a restart.

Activating a scene

Calling scene.turn_on issues PUT /api/v2/device/apps/:package/widgets/:id/activate, which brings the target widget to the foreground. No action is triggered.

activate_action entity service

Use the activate_action entity service to additionally trigger a widget-specific action. This calls POST /api/v2/device/apps/:package/widgets/:id/actions. See the full reference on the Entity Services page.

Entity metadata

Attribute Value
Unique ID <serial>-<package>-<widget_id>
Name App title as reported by the device
is_active attribute true when this widget is currently in the foreground

See the Apps API reference for endpoint details.

Select non-SKY

GET /api/v2/device/display API 2.0.0+
PUT /api/v2/device/display API 2.0.0+

One select entity controls the display brightness mode on non-SKY devices. SKY devices do not expose a brightness_mode field.

Entity key Options API field
brightness_mode
auto manual
display.brightness_mode
  • auto — The device adjusts brightness based on its ambient light sensor. Setting the brightness number entity while in this mode is accepted by the API but takes effect only when the mode is switched to manual.
  • manual — Brightness is fixed to the value in the brightness number entity.
Screensaver target The brightness_mode select entity is also the target entity for the set_screensaver entity service. See the Entity Services page.

See the Display API reference.

Sensor

GET /api/v2/device API 2.0.0+

One diagnostic sensor entity, disabled by default, exposes the Wi-Fi signal strength.

Entity key Unit State class Category Default
rssi % measurement Diagnostic Disabled

The value is read from wifi.signal_strength in the device state object returned by GET /api/v2/device. It is expressed as a percentage (0–100) rather than dBm. Enable the entity in the entity registry if you want to include it in dashboards or automations.

See the Device State API reference.

Switch

PUT /api/v2/device/display API 2.0.0+
PUT /api/v2/device/bluetooth API 2.0.0+

Up to two switch entities are created per device, conditional on the capabilities reported by the device state.

Entity key Controls Availability gate API field
display_on Turn the display on or off display.on is not None PUT /api/v2/device/display → {"on": bool}
bluetooth_active Enable or disable Bluetooth bluetooth.available is true PUT /api/v2/device/bluetooth → {"active": bool}

Both entities are placed in the Configuration entity category.

See the Display and Bluetooth API references.

Text

PUT /api/v2/device/bluetooth API 2.0.0+

One text entity is created when Bluetooth is reported as available. It sets the Bluetooth broadcast name of the device.

Entity key Min length Max length Mode API field
bluetooth_name 1 248 text bluetooth.name → PUT /api/v2/device/bluetooth → {"name": "…"}

The entity is placed in the Configuration category. The name change is broadcast immediately via Bluetooth. See the Bluetooth API reference.

Update

GET /api/v2/device API 2.3.0+

One firmware update tracking entity is registered for devices running OS 2.3.0 or later (the first version that exposes update information via the local API). Devices on older firmware do not get this entity.

Property Source Notes
installed_version device_state.os_version Current OS version string, e.g. 2.3.0
latest_version device_state.update.version Newest available version; equals installed_version when up to date
Device class — FIRMWARE
Install action The entity reports available updates but does not initiate firmware installation — firmware updates are managed through the LaMetric mobile app. The entity is informational only.

See the Device State API reference.
